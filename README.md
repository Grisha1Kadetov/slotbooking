# Room Slots Booking Service
Сервис предназначен для бронирования переговорок:
- **admin** создаёт переговорки и расписания;
- **user** смотрит доступные слоты, создаёт и отменяет брони;
- система сама формирует слоты по расписанию.

---

## Что реализовано

### Функционал
- создание переговорок (**admin**);
- создание расписания переговорки **один раз** (**admin**);
- автоматическая генерация слотов длительностью **30 минут**;
- просмотр списка переговорок (**admin**, **user**);
- просмотр доступных слотов по переговорке и дате (**admin**, **user**);
- создание брони на слот от имени пользователя (**user**);
- отмена своей брони (**user**);
- просмотр списка своих будущих броней (**user**);
- просмотр списка всех броней с пагинацией (**admin**);
- регистрация и логин

Полный список контрактов можно посомтреть с помощью ***api.yml***

### Тесты
- юнит-тесты на бизнес-сценарии;
- E2E тест на сценарий: 
  - создание переговорки → создание расписания → создание брони → отмена брони;

Покрытие проекта тестами: **50%~**.

## Архитектура

Проект разбит на привычные слои:
- `handler` — HTTP, DTO, валидация входных данных, преобразование ошибок в HTTP-коды;
- `service` — бизнес-логика;
- `repository` — работа с БД;
- `middleware` — JWT авторизация;
- `model` — доменные сущности и бизнес-ошибки.

### Как формируются слоты

**Гибридный подход**:
- при создании расписания слоты предгенерируются на ближайшее окно дат;
- при запросе доступных слотов на конкретный день, если слоты на этот день ещё не существуют, они создаются лениво.

#### Почему не генерировать только «по запросу»
Если делать слоты исключительно лениво, то первый запрос на новый день будет тяжелее: нужно сначала получить расписание, потом сформировать слоты, потом записать их в БД. Это ухудшает latency эндпоинта.

#### Примечание
Я также задумывался об использовании Redis для этой ручки, если бы нагрузка была бы большой, то я бы добавил кэш на получение комнат, при резервации удалял и генерировал новый.
